## 현재 상태 (사실 확인)
- 인제스트 실제 완주 확인: 추출 190s · 표준화 50s · 라벨표준화 181s · 중복 2s · 플래그 127s · 검수큐 60s ≈ **약 10분**(2023 PDF 1개 기준).
- 검수 큐 289행(미검수 high 33). 라이브 인덱스 198청크(게이트 미통과 데이터로 구축됨 — 아래 D5).
- 동작은 하지만 (1) 인제스트가 느리고 (2) 각 단계가 무슨 작업인지/다음에 뭘 할지 안내가 부족하고 (3) 검수에서 "값 없는 행"을 사용자가 직접 찾아야 함.

## D1. 인제스트 속도 최적화

**관찰**: 약 10분. 원인은 **표준화 이후 단계가 매번 전 연도를 LLM으로 재처리**(`standardize`가 `outputs/*.extracted.jsonl` 전체 글로빙)하기 때문. PDF 1개만 추가/수정해도 2023·2024·2025 전체를 다시 돌림.

**선택지**
- **A. 변경 없으면 단계 스킵(캐시)**: 입력 파일이 그대로면 산출물 재사용(타임스탬프/해시 비교). 재실행 시간 대폭 절감. "강제 재실행" 옵션 유지. → *중간 규모, 안전.*
- **B. LLM 배치 호출**: `extract`가 지금 블록당 1콜 → 배치로 묶음(`refine`/`flags`도). 속도↑, 정확도 동일. → *중간 규모.*
- **C. 증분 처리**: 바뀐 연도만 `refine`/`flags` 재실행. 표준화 사전이 누적식이라 까다롭고 위험. → *큰 변경, 비추천(지금은).*
- **D. 그대로 두기**: 가끔 도는 배치로 보고 최적화 안 함, 진행 표시만 유지. → *0 작업.*

**제 추천**: **A(스킵 캐시)** 먼저. 가장 효과 크고 안전. 반복 실행이 잦으면 **B**도. (C는 나중.)
**결정 시 할 일**: `pipeline.py`에 "입력이 최신이면 스킵" 로직 + 단계별 "스킵됨/재실행" 표시. (B 선택 시 extract 배치화.)
**검증**: 같은 데이터로 2회 실행 → 2회차가 LLM 단계 스킵으로 수초 내 완료. E2E로 "스킵" 표시 확인.

## D2. 각 단계 "무슨 작업인지" 설명 + "다음 행동" 안내 (UX)

**관찰**: 단계 상단에 그 단계가 무엇을/왜 하는지, 끝나면 다음에 뭘 눌러야 하는지 안내가 약함.

**선택지**
- **A. 전 단계 적용**: 각 단계 상단에 1~2줄 설명("이 단계는 X를 합니다 · 보통 Y분 · 끝나면 Z") + 완료 후 **"다음: N단계로 가기" 버튼**. → *작고 안전, 추천.*
- **B. 일부 단계만**.

**제 추천**: **A**. 저위험 UX 개선.
**결정 시 할 일**: 5개 단계에 설명 헤더 + 완료 시 다음-단계 이동 버튼. 사이드 "다음 할 일" 안내와 일관.
**검증**: Playwright로 각 단계 설명 텍스트·다음 버튼 노출 확인.

## D3. 검수 안내 — "값 없는 행"을 사용자가 일일이 찾지 않게 ★사용자 핵심 지적★

**관찰**: 검수 큐 289행에서 **기본값·정정값이 모두 없는(빈 값) 행**을 사용자가 직접 찾아야 함. 부적절.

**선택지**
- **A. 상단 배너 + 기본 필터**: "⚠️ 값이 비어 반드시 검수 필요: N행" 배너 + **"값 없는 행만 보기" 필터(기본 ON)**. 못 찾는 문제 해소.
- **B. 정렬**: 값 없는 행을 표 맨 위로.
- **C. 순차 안내**: "다음 미처리 행으로 →" 버튼으로 하나씩 이동.
- **D. 게이트 연동**: `validate.py`의 차단 항목(빈값·미확정 비전·미검수 high)을 검수 탭 상단 **"할 일 목록"**으로 띄우고, 클릭하면 그 행들만 필터. (게이트 ↔ 검수 UI 연결)

**제 추천**: **A + B**(배너+필터+정렬)로 즉시 해결, 여유 되면 **D**(게이트 연동)로 "무엇을 검수해야 인덱싱 가능한지" 한눈에. C는 선택.
**결정 시 할 일**: 검수 탭에 빈값 카운트 배너 + "값 없는 행만" 필터/정렬. (D 선택 시 게이트 항목→필터 버튼.)
**검증**: Playwright로 배너·필터 동작, 빈값 행이 먼저 보이는지 확인.

## D4. 흐름 범위 — 스텝퍼 유지? (이전부터 미결)

**배경**: 원래 작은 UX 요청을 제가 전면 스텝퍼로 키웠음. 다만 지금은 동작·완주 확인됨.

**선택지**
- **A. 현 스텝퍼 유지** + D2/D3로 안내 보강. → *추천(이미 동작, 흐름 명확).*
- **B. 최소안 축소**: 탭 구조 복귀 + 로깅/진행만. → *방금 만든 것 상당 폐기.*
- **C. 절충**: 스텝퍼 유지하되 인제스트 등 고급 기능 접기.

**제 추천**: **A**.

## D5. 인덱스–게이트 정합 (원칙 위반 해소, 백로그 A)

**관찰**: 라이브 인덱스(198청크)가 게이트 미통과 데이터(빈값·미확정 비전 38·미검수 high 33)로 구축됨 → "추측은 데이터 아님" 원칙을 실제로는 위반 중.

**선택지**
- **A. 검수로 확정 후 재인덱싱**: D3로 검수 → 게이트 통과 → 재인덱싱. *정공법(시간 듦).*
- **B. 경고만**: 앱에 "인덱스가 미확정 데이터 포함/게이트 미통과" 경고 + 신선도 표시. *빠른 임시.*
- **C. A+B**: 경고 먼저 띄우고, 검수 진행하며 점차 정합.

**제 추천**: **C** — 지금 경고 표시(정직성) + D3 검수로 점진 정합 후 재인덱싱.

**관찰**: E2E가 전부 `RAG_FAKE_LLM`이라 UI 배선만 증명. 실제 검색/답변/로그/인제스트 미검증.

**선택지**

**제 추천**: **A 먼저**(저비용), 이후 **B**.
